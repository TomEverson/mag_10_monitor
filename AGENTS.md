# AGENTS.md

## Project Overview

**mag10-monitor** is a real-time market intelligence pipeline built on GCP.
It connects to the Finnhub WebSocket feed, detects market signals across
MAG 10 stocks, and surfaces them on a Looker Studio dashboard.

This project follows **spec-driven development**. Before writing any
implementation code, read the relevant spec file(s) in `/spec`. The spec
defines the contract вЂ” the code implements against it. If the spec and the
code disagree, the spec wins unless a spec change is explicitly approved.

## Spec-Driven Workflow

Every feature follows this order вЂ” do not skip steps:

1. **Read the spec** вЂ” find the relevant file under `/spec` before touching code
2. **Clarify before implementing** вЂ” if the spec is ambiguous or missing detail, ask before guessing
3. **Implement against the spec** вЂ” code should satisfy exactly what the spec defines, nothing more
4. **Update the spec if behaviour changes** вЂ” never silently diverge from the spec

If a spec file does not exist yet for the task at hand, create it first and
get confirmation before writing implementation code.

## Service Responsibilities

### listener/
- Owns the Finnhub WebSocket connection
- Responsible for reconnection logic and graceful shutdown
- Runs all 4 detectors concurrently вЂ” each detector is stateful and holds
  its own rolling window in memory
- Only responsibility toward Pub/Sub is publishing вЂ” it does not read from it
- Must not write directly to BigQuery or GCS вЂ” only publishes events

### functions/
- Each function is triggered by exactly one Pub/Sub topic
- Each function handles one concern only вЂ” do not combine handlers
- Writes to BigQuery via streaming insert using `shared/bq_client.py`
- Archives raw payload to GCS before any transformation
- Must be idempotent вЂ” reprocessing the same event must not create duplicates

### infra/
- All GCP resources are defined here вЂ” never create resources manually in console
- Each resource type has its own module
- Naming convention: `mag10-{resource-type}-{env}` (e.g. `mag10-pubsub-prod`)
- Secrets are managed via GCP Secret Manager вЂ” never hardcoded or in .env files committed to git

### dashboard/queries/
- Each `.sql` file corresponds to one Looker Studio board
- Queries must reference partitioned BigQuery tables using `WHERE DATE(timestamp) = @date`
  to avoid full table scans

## Key Constraints

- **Finnhub WebSocket**: 50 symbol subscription limit on free tier вЂ” currently using 10
- **Market hours**: WebSocket delivers trades only during US market hours
  (9:30amвЂ“4:00pm ET, MonвЂ“Fri). The listener must handle the no-trade window
  without crashing or reconnecting in a loop
- **BigQuery streaming inserts**: billed per MB вЂ” keep payloads lean, no
  redundant fields
- **Cloud Functions**: stateless вЂ” all rolling window state lives in the
  listener, never in functions
- **VM**: e2-micro вЂ” keep listener memory footprint minimal; rolling windows
  should have a maximum length defined in `config.py`

## Package Management

This project uses **uv** exclusively. Do not use `pip` directly.

### Common commands

```bash
# Install dependencies from lockfile
uv sync

# Add a new dependency
uv add <package>

# Add a dev-only dependency
uv add --dev <package>

# Run a script inside the venv
uv run python main.py

# Generate requirements.txt for GCP Cloud Function deployment
uv pip compile pyproject.toml -o requirements.txt
```

### Rules

- `uv.lock` is always committed вЂ” never gitignored
- `pyproject.toml` is the source of truth for dependencies вЂ” never edit `requirements.txt` by hand
- `requirements.txt` inside each `functions/*/` directory is **generated only**, used solely for GCP deployment. Regenerate it after any dependency change with `uv pip compile`
- Each service (`listener/`, `functions/volume/`, etc.) has its own isolated `pyproject.toml` and `uv.lock` вЂ” do not share a root-level lockfile across services

## What Not To Do

- Do not use `pip` directly вЂ” always use `uv`
- Do not edit `requirements.txt` by hand вЂ” regenerate with `uv pip compile`
- Do not write implementation code before reading the relevant spec
- Do not create GCP resources manually вЂ” use Terraform in `infra/`
- Do not hardcode symbols, thresholds, or topic names вЂ” all go in `config.py`
- Do not import across service boundaries (listener importing from functions or vice versa)
- Do not write directly to BigQuery from the listener вЂ” only publish to Pub/Sub
- Do not commit `.env` files or any file containing credentials
- Do not combine multiple detector concerns into a single Cloud Function
